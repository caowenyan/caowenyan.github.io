ajax跨域

什么是跨域

协议、域名、端口

产生跨域问题的原因

  1. 浏览器限制
  2. 跨域（违反同源策略）
    所有的浏览器都支持同源策略（解决安全问题，针对cookie提出的），所谓的同源是指三个相同：
    • 协议相同
    • 域名相同
    • 端口相同
  3. XHR(XMLHttpRequest)请求

解决思路

  • 浏览器（客户端可以改动，不可行）

  • xhr（JSONP）
    现在简单的介绍一下jsonp

    • jsonp是一种动态插入script，执行完就删除插入的代码
    • 若是有基本的ajax改为jsonp请求，会报错误Uncaught SyntaxError: Unexpected token :
    • 所以需要服务器后台也需要改动
    • jsonp是一种约定，callback是默认值，也可以改变约定的名字

    • jsonp和ajax的对比

      请求类型|type|Content-type|url
      —|:–:|:–:|:–:|—:
      jsonp|script|application/javascript|uri?callback=…..
      ajax|xhr|application/json|uri

    • jsonp弊端
      • 需要服务器改动代码
      • 只支持get请求
      • 发送的不是xhr请求（不支持异步、事件等）
  • 跨域
    • 被调用方：支持跨域，修改代码，同公司
      • 服务器实现
        • 浏览器是先执行还是先判断？
          • 简单请求是先请求在判断，非简单请求是先判断（OPTIONS）在请求
          • 简单请求和非简单请求对比：
            1. 方法： GET、HEAD、POST
            2. header
              • 无自定义头
              • Content-Type：
                • text/plain
                • multipart/form-data
                • application/x-www-form-unlencoded
                  不满足以上2点的都是非简单请求，如方法为DELETE，Content-Type为application/json，增加header保存一些共有信息。
                  *参数讲解
          • Access-Control-Allow-Origin：浏览器的地址，也可以是*，但是针对cookie不可取
          • Access-Control-Allow-Method：支持的方法，加入DELETE等或者“*”
          • Access-Control-Allow-Header：加入Content-Type，还有前端主动加入的header字段，也可以通过获取request的Access-Control-Allow-Header字段填充response的header
          • Access-Control-Max-Age：预检命令缓存的时间（秒），在缓存的时间内即使后端的cors设置错误，也不会影响正常访问，除非清除缓存，或者预检缓存到期
          • Access-Control-Allow-Credentials：cookie应该加到被调用方，因为是在被调用方获取cookie的
      • nginx实现
        • 虚拟主机：多个域名指向同一个服务器
        • 在conf目录下简历vhost目录，在nginx.conf最后加入include vhost/*.conf
          *配置文件例子
  • apache实现
    参考https://www.imooc.com/video/16592
  • 调用方：隐藏跨域
    • 反向代理：访问a，其实访问的是b
    • host和conf配置

  4. 创建github仓库，名称必须是你的用户名.github.io，将来访问的地址是：http://你的用户名.github.io
